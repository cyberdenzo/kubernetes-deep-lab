# Lab Rules

Правила лабораторной работы в Kubernetes Deep Lab: Cohort 1.

## Главный принцип

Kubernetes Deep Lab - это не пассивный курс, а инженерная лаборатория.

Цель - не просто повторить команды, а понять, как Kubernetes работает внутри, как он ломается, как диагностируется и как восстанавливается.

## Формат работы

Участник должен:

- выполнять лабораторные задания
- вести lab journal
- фиксировать команды и выводы
- описывать ошибки и способы решения
- задавать вопросы с контекстом
- доводить задания до проверяемого результата

## Что считается участием

Участие - это активная работа.

Активный участник:

- выполняет задания
- присылает отчеты
- открывает Pull Request
- отвечает на review
- показывает выводы команд
- разбирает ошибки
- помогает улучшать материалы

Пассивное наблюдение не считается участием.

## Бесплатный формат

Участие бесплатное.

Но бесплатность не означает пассивный режим. Место в группе занимает тот, кто реально работает и двигается по лабораторным.

Если участник системно не выполняет задания и не выходит на связь, его место может быть освобождено для другого кандидата.

## Ритм

Базовый ритм:

- одна лабораторная в неделю
- один отчет на лабораторную
- один Pull Request или обновление lab journal
- вопросы задаются по мере выполнения
- ошибки фиксируются как часть обучения

## Как задавать вопросы

Плохой формат:

    Не работает.

Хороший формат:

    Что делал:
    - команда
    - ожидаемый результат
    - фактический результат

    Вывод:
    <вставить текстовый вывод команды>

    Что уже проверил:
    - ...

## Выводы команд

По возможности выводы команд нужно присылать текстом, а не скриншотами.

Скриншоты допустимы, если:

- проблема связана с UI
- текстовый вывод невозможно удобно скопировать
- нужно показать визуальное состояние

## Ошибки

Ошибки - это нормальная часть лаборатории.

Но каждую ошибку нужно разбирать инженерно:

- какой слой затронут
- какая команда выполнялась
- какой вывод получен
- что ожидалось
- что уже проверено
- какой следующий диагностический шаг

## Диагностический подход

Сначала проверяем состояние, потом меняем.

Не нужно сразу переустанавливать кластер, пересоздавать VM или менять несколько параметров одновременно.

Правильный порядок:

1. Зафиксировать симптом.
2. Проверить текущий state.
3. Локализовать слой.
4. Сделать минимальное изменение.
5. Проверить результат.
6. Записать вывод в journal.

## Рискованные действия

Перед рискованными действиями нужно понимать последствия.

Примеры рискованных действий:

- удаление VM
- сброс Kubernetes-ноды
- удаление etcd data
- изменение сетевых правил
- переустановка container runtime
- пересоздание кластера
- удаление namespace с workload

Для таких действий нужно явно указать:

- риск
- что будет изменено
- как проверить до действия
- как откатиться
- как проверить после действия

## Запрет на преждевременную автоматизацию

Не автоматизируем то, что еще не поняли вручную.

Порядок:

1. Сначала ручное понимание.
2. Потом повторяемость.
3. Потом минимальная автоматизация.
4. Потом Ansible/IaC.

## Чистые VM

До соответствующей лабораторной не нужно заранее устанавливать:

- Kubernetes
- kubeadm
- kubelet
- kubectl
- CNI
- Helm
- Argo CD
- готовые Kubernetes-дистрибутивы

Нам нужны чистые VM или возможность быстро пересоздать чистые VM.

## GitHub workflow

Базовый workflow:

1. Участник делает fork repository.
2. Создает свою папку students/<github-username>/.
3. Заполняет environment.md.
4. Открывает Pull Request.
5. Получает review.
6. Вносит правки, если нужно.
7. После acceptance продолжает вести lab reports.

## Lab journal

Lab journal нужен не для отчетности ради отчетности.

Он помогает:

- видеть прогресс
- фиксировать ошибки
- собирать troubleshooting-базу
- превращать практику в публичный материал
- готовиться к собеседованиям
- не терять контекст

## Минимальный lab report

Каждый отчет должен содержать:

- цель лабораторной
- что было сделано
- какие команды выполнялись
- ключевые выводы команд
- какие ошибки возникли
- как ошибки были исправлены
- что стало понятнее
- какие вопросы остались

## Единый lab path

Участники могут иметь разный опыт и разные стенды, но основная лабораторная траектория должна быть общей.

Это нужно, чтобы:

- проще сравнивать результаты
- проще разбирать ошибки
- не превращать группу в хаотичный support-chat
- сохранить единый темп первой когорты

## Источники

Основной источник истины:

- официальная документация Kubernetes
- официальная документация используемых компонентов
- фактический вывод команд в лабораторном стенде

Medium, random blogs и старые инструкции не считаются источником истины.

## Коммуникация

Ожидается взрослая инженерная коммуникация:

- писать конкретно
- не скрывать ошибки
- не бояться задавать вопросы
- не спорить с выводом команд
- не перепрыгивать через шаги
- уважать время других участников

## Критерий успешного участия

Участник успешно проходит формат, если он:

- регулярно выполняет лабораторные
- ведет lab journal
- показывает реальные выводы команд
- понимает, что делает
- умеет объяснить результат
- двигается от ручного понимания к инженерной уверенности
